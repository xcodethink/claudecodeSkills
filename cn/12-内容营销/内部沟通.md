# 内部沟通助手（Internal Comms）

一套帮助撰写各类内部沟通的资源，使用公司常用的格式。当需要撰写内部沟通文档时（状态报告、领导层更新、3P 更新、公司新闻通讯、FAQ、事件报告、项目更新等），Claude 应使用此技能。

## 适用场景

- 3P 更新（Progress / Plans / Problems，即进展/计划/问题）
- 公司新闻通讯
- FAQ（常见问题）解答
- 状态报告
- 领导层更新
- 项目更新
- 事件报告

## 使用方法

撰写任何内部沟通时：

1. **识别沟通类型**
2. **加载相应的指南文件**：
   - 3P 更新 → 见下方"3P 更新指南"
   - 公司新闻通讯 → 见下方"公司新闻通讯指南"
   - FAQ 解答 → 见下方"FAQ 解答指南"
   - 其他类型 → 见下方"通用沟通指南"
3. **按照该文件中的具体说明**进行格式化、语调和内容收集

如果沟通类型不匹配任何现有指南，请要求澄清或了解更多所需格式的上下文。

**关键词**：3P updates、company newsletter、company comms、weekly update、faqs、common questions、updates、internal comms

## 一、3P 更新指南

### 说明

3P 更新代表 "Progress, Plans, Problems（进展、计划、问题）"。主要受众是高管、领导层和其他团队成员。它们应该非常简洁和直截了当：阅读时间控制在 30-60 秒以内。读者对团队工作有一些但不多的背景了解。

3P 可以覆盖任何规模的团队，从小组到整个公司。团队越大，任务描述应越宏观。例如，"移动团队"可能包含"发布了功能"或"修复了 bug"，而公司级别可能有更重大的 3P，如"招聘了 20 名新员工"或"签下了 10 个新客户"。

它们代表团队在一个时间段（几乎总是一周）的工作，包含三个部分：

1. **Progress（进展）**：团队在过去的时间段内完成了什么。主要关注已发布的东西、达成的里程碑、创建的任务等
2. **Plans（计划）**：团队在下一个时间段计划做什么。关注团队最关注的、真正高优先级的事项
3. **Problems（问题）**：任何拖慢团队进度的事情。可能是人手不足、阻碍前进的 bug 或障碍、某个交易落空等

### 工具可用

尽可能从可用来源获取信息：
- **Slack**：团队成员的更新帖子——寻找大频道中有很多反应的帖子
- **Google Drive**：关键团队成员撰写的高浏览量文档
- **Email**：有大量回复或大量相关内容的邮件
- **Calendar**：重要的非定期会议，如产品评审等

信息收集的时间范围：
- Progress：一周前到今天
- Plans：今天到下一周
- Problems：一周前到今天

### 工作流

1. **确认范围**：确认团队名称和时间段
2. **收集信息**：使用可用工具或直接询问用户
3. **起草更新**：遵循严格的格式化指南
4. **审查**：确保简洁（30-60 秒阅读）且数据驱动

### 格式

格式始终相同，非常严格。绝不使用其他格式。选择一个有趣且能捕捉团队和更新氛围的 emoji。

```
[选择一个 emoji] [团队名称]（覆盖日期，通常一周）
Progress: [1-3 句内容]
Plans: [1-3 句内容]
Problems: [1-3 句内容]
```

每个部分不超过 1-3 句：清晰、直截了当。应该数据驱动，尽可能包含指标。语调应该非常客观，不要过于修辞化。

## 二、公司新闻通讯指南

### 说明

撰写公司级别的新闻通讯更新。总结公司过去一周/一个月的情况，以新闻通讯的形式让全公司阅读。大约 20-25 个要点。将通过 Slack 和邮件发送。

理想特征：
- **多链接**：从 Google Drive 拉取相关文档、链接到公告频道中显著的 Slack 消息、引用全公司的邮件、突出公司发生的重要事件
- **简短直接**：每个要点不超过约 1-2 句
- **使用"我们"的语调**：作为公司的一份子。许多要点应该说"我们做了这个"或"我们做了那个"

### 可用工具
- **Slack**：在有很多人、很多反应或很多讨论的频道中查找消息
- **Email**：高管讨论全公司公告的邮件
- **Calendar**：大型出席者名单的会议（如 All-Hands 全员会议），附带的文档是很好的链接
- **Documents**：最近一两周内发布的、获得大量关注的新文档
- **External press（外部媒体）**：过去一周的文章或媒体报道

### 分节

公司很大（1000+ 人），有各种不同的团队和举措。为确保更新效果好，按相似主题分节。可能的分组方式：
- {产品开发, 市场推广, 财务}
- {招聘, 执行, 愿景}
- {外部新闻, 内部新闻}

确保公司不同领域都得到充分体现。

### 优先级

**关注**：
- 全公司影响（而非团队级别细节）
- 领导层公告
- 重大里程碑和成就
- 影响大多数员工的信息
- 外部认可或媒体报道

**避免**：
- 过于细粒度的团队更新（留给 3P）
- 仅与小群体相关的信息
- 已经传达过的重复信息

### 示例格式

```
:megaphone: 公司公告
- 公告 1
- 公告 2
- 公告 3

:dart: 优先事项进展
- 领域 1
    - 子项 1
    - 子项 2
    - 子项 3
- 领域 2
    - 子项 1
    - 子项 2

:pillar: 领导层更新
- 帖子 1
- 帖子 2

:thread: 社交动态
- 更新 1
- 更新 2
```

## 三、FAQ 解答指南

### 说明

你是一个回答全公司常见问题的助手。每周全公司都有很多问题被提出，你的目标是总结这些问题。我们希望公司信息透明、认知一致，所以你的工作是产生一组员工常问的问题并尝试回答。

两个核心任务：
- **发现问题**：找到让大量员工困惑的大问题，通常是影响大部分员工群体的事项
- **提供答案**：尝试给出总结性的答案以减少困惑

可能感兴趣的领域示例：近期公司事件（融资、新高管等）、即将发布的产品、招聘进展、愿景或重点的变化等。

### 可用工具
- **Slack**：全公司范围内的问题——可能是对有大量回复的帖子的提问、有很多反应或点赞表示支持的问题
- **Email**：直接包含 FAQ 的邮件
- **Documents**：Google Drive 中的文档、日历事件中链接的文档

### 格式

```
- *Question*: [插入问题 - 1 句话]
- *Answer*: [插入答案 - 1-2 句话]
```

### 指导原则
- 确保问题全面，不要只关注用户所在的团队
- 尽可能全面地阅读所有可用工具
- 基于官方公司沟通回答
- 如果信息不确定，明确指出
- 链接权威来源（文档、公告、邮件）
- 保持语调专业但平易近人
- 如果问题需要高管意见或官方回复，标记出来

## 四、通用沟通指南

### 说明

撰写不属于标准格式（3P 更新、新闻通讯或 FAQ）的内部公司沟通。

在开始之前：
1. 询问用户目标受众
2. 理解沟通的目的
3. 澄清期望的语调（正式、随意、紧急、信息性）
4. 确认任何具体的格式要求

### 通用原则
- 清晰简洁
- 使用主动语态
- 将最重要的信息放在最前面
- 包含相关链接和引用
- 匹配公司的沟通风格
