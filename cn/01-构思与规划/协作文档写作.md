# 协作文档写作（Doc Co-authoring）

## 概述

Anthropic 官方的协作文档写作工作流。三阶段结构化流程，从理解上下文到迭代优化到读者测试，确保产出高质量文档。

## 三阶段工作流

### 阶段 1：上下文收集（Context Gathering）

**目标**：在动手写之前，充分理解需求。

1. **元上下文提问**
   - 这是什么类型的文档？（技术方案/博客/内部汇报...）
   - 目标读者是谁？他们的背景知识？
   - 这份文档的预期影响是什么？
   - 格式和长度要求？

2. **信息倾泻（Info Dump）**
   - 请用户将所有相关上下文一次性提供
   - 不要打断——让信息自然流出
   - 标记需要后续澄清的点

3. **澄清提问**
   - 针对模糊点提出具体问题
   - 闭合理解差距
   - 确认关键假设

4. **退出条件**：当你对文档需求有了足够的理解，进入阶段 2。

### 阶段 2：优化与结构化（Refinement & Structure）

**目标**：逐章节构建高质量文档。

对每个章节执行以下循环：

```
澄清问题 → 头脑风暴 (5-20 个选项) → 筛选最佳 → 查漏补缺 → 起草 → 迭代优化
```

| 步骤 | 说明 |
|------|------|
| 澄清问题 | 确认本章节的具体要求 |
| 头脑风暴 | 为每个段落/要点生成 5-20 个候选内容 |
| 筛选 | 选出最有价值的内容 |
| 查漏 | 检查是否遗漏关键信息 |
| 起草 | 写出初稿 |
| 迭代 | 连续 3 次无修改时，进入下一章节 |

**全文检查**：所有章节完成后，通读全文检查：
- 章节间的逻辑流是否顺畅？
- 术语使用是否一致？
- 是否有冗余内容？

### 阶段 3：读者测试（Reader Testing）

**目标**：用"新鲜的眼睛"发现盲点。

1. **预测读者问题**
   - 读者看到这份文档后最可能提出什么疑问？
   - 列出 3-5 个预测问题

2. **子代理测试**（如果可用）
   - 派遣一个新的 Claude 实例阅读文档
   - 收集它的问题和困惑点
   - 这些就是你的盲点

3. **额外检查**
   - 是否有歧义表述？
   - 是否有内部矛盾？
   - 非领域专家能否理解？

4. **基于测试结果迭代**
   - 修复发现的问题
   - 必要时回到阶段 2 重写特定章节

## 使用技巧

- **有效引导**：不需要事事同意 Claude，直接说"这部分不对，改成这样"
- **处理偏离**：如果写作方向偏了，直接指出要回到哪个点
- **上下文管理**：文档变长时，定期总结当前状态
- **制品管理**：在文件系统中保存中间版本，便于回退
