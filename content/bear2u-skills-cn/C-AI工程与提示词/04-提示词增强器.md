# 提示词增强器（Prompt Enhancer）

将简短的开发请求转化为清晰详细的需求规格，通过分析项目上下文（代码结构、依赖、规范、现有模式）生成更准确的、上下文感知的提示词。在实施前将增强后的需求呈现给用户确认。

## 使用场景

在以下情况使用此技能：
- 用户提供简短的开发请求，如"帮我做登录功能"、"加个 API"
- 请求缺少具体实现细节
- 用户上传了项目文件或提到"这个项目"
- 任务需要理解项目架构

## 核心工作流程

### 第一步：分析项目上下文

**检查上传文件：**
```bash
view /mnt/user-data/uploads
```

**采集关键信息：**
- 项目结构和组织方式
- 技术栈（package.json、pubspec.yaml、requirements.txt 等）
- 现有模式（状态管理、API 调用、路由）
- 代码规范（命名、文件结构）
- 类似的已有功能

### 第二步：提取请求意图

从用户的简短请求中识别：
- **功能类型**：新功能、Bug 修复、重构、API 集成
- **范围**：单一页面、完整流程、后端 + 前端
- **依赖**：相关功能或系统

### 第三步：构建增强需求

创建结构化需求文档：

```markdown
# [功能名] 实现需求

## 项目上下文
- 框架：[检测到的框架和版本]
- 架构：[检测到的模式]
- 状态管理：[检测到的库]
- 关键库：[列出相关依赖]

## 实现范围

### 主要功能
1. [主功能 1]
2. [主功能 2]
3. [主功能 3]

### 文件结构
```
[基于项目预期的文件结构]
```

## 详细需求

### 1. [层/组件名称]
- **位置**：[文件路径]
- **用途**：[做什么]
- **实现内容**：
  - [具体需求 1]
  - [具体需求 2]
- **遵循现有模式**：[引用已有模式]

### 2. [下一个层/组件]
...

## 成功标准
- [ ] [验收标准 1]
- [ ] [验收标准 2]
- [ ] [验收标准 3]
- [ ] 保持现有代码风格和架构一致性
- [ ] 为所有主要功能编写测试

## 待确认事项
- [需要澄清的问题]
- [做出的假设]

---
按这个需求进行吗？有需要修改的地方请告诉我。
```

### 第四步：呈现给用户

**重要**：创建增强需求后，呈现给用户并请求确认：

```
以上是根据分析整理的需求。

可以按这个方案进行吗？
有需要修改或补充的内容请告诉我！
```

**不要在用户确认之前开始实施。** 目标是先澄清需求。

## 各技术栈分析模式

### Flutter 项目

**检测标志**：pubspec.yaml、lib/ 目录

**需采集的关键上下文：**
- 状态管理（Riverpod、Bloc、Provider、GetX）
- 架构（Clean Architecture、MVVM、MVC）
- 导航（go_router、auto_route、Navigator）
- 网络（Dio、http）
- 本地存储（Hive、SharedPreferences、SQLite）

**增强需求应包含：**
```markdown
## 实现范围

### 表现层（Presentation Layer）
- 页面：lib/presentation/[feature]/[screen]_screen.dart
- 状态：[StateNotifier/Bloc/Controller] 搭配 [状态模式]
- 组件：可复用组件

### 领域层（Domain Layer）
- 实体：lib/domain/entities/[name].dart
- 用例：lib/domain/usecases/[action]_usecase.dart
- 仓库接口：lib/domain/repositories/

### 数据层（Data Layer）
- 模型：lib/data/models/[name]_model.dart（fromJson/toJson）
- 仓库实现：lib/data/repositories/
- 数据源：lib/data/datasources/

### 导航
- 路由：[路由路径]
- 导航方式：[基于路由的 context.go/push]

## 成功标准
- [状态管理]管理状态
- 保持与 [现有组件] 的样式一致性
- API 响应错误处理
- 显示加载状态
- 编写 Widget 测试
```

### Next.js/React 项目

**检测标志**：package.json 中包含 "next" 或 "react"

**需采集的关键上下文：**
- Next.js 版本（App Router vs Pages Router）
- 状态管理（Zustand、Redux、Context）
- 样式方案（Tailwind、CSS Modules、styled-components）
- API 方式（Next.js API routes、外部 API）
- TypeScript 使用情况

**增强需求应包含：**
```markdown
## 实现范围

### UI 组件
- 组件：[path]/[ComponentName].tsx
- Props 接口：[现有模式]
- 样式：[Tailwind 类/CSS modules]

### 状态管理
- Store：[现有状态结构]
- Actions：[动作命名规范]

### API 层
- 端点：[app/api 或 外部]
- 方法：[GET/POST/PUT/DELETE]
- 响应类型：[TypeScript 接口]

### 路由
- 路由：[app/[route] 或 pages/[route]]
- 动态段：[如需要]

## 成功标准
- TypeScript 类型安全
- 复用 [现有组件]
- 响应式设计（移动端/桌面端）
- SEO 优化（元数据）
- 单元测试 + 集成测试
```

### Python（Django/FastAPI）项目

**检测标志**：requirements.txt、manage.py、main.py

**需采集的关键上下文：**
- 框架（Django、FastAPI、Flask）
- ORM（SQLAlchemy、Django ORM、Tortoise）
- 认证方式
- API 文档（OpenAPI/Swagger）

**增强需求应包含：**
```markdown
## 实现范围

### 模型/Schema
- 模型：[app]/models.py 或 schemas/[name].py
- 字段：[带类型的字段定义]
- 关系：[外键/关联]

### 视图/端点
- 路由：[路径]
- 方法：[HTTP 方法]
- 请求/响应 Schema：[Pydantic 模型]

### 业务逻辑
- 服务：[如适用]
- 验证：[输入验证]

### 数据库
- 迁移：[迁移命令]

## 成功标准
- API 文档自动生成
- 请求验证
- 带正确状态码的错误处理
- 单元测试 + 集成测试
- 应用 [认证方式]
```

## 增强示例

**用户输入：**
```
帮我实现登录功能
```

**技能输出：**
```markdown
# 登录功能实现需求

## 项目上下文
- 框架：Flutter 3.16.0（null safety）
- 架构：Clean Architecture
- 状态管理：Riverpod（StateNotifier 模式）
- 网络：Dio 5.3.0
- 存储：flutter_secure_storage
- 导航：go_router

## 实现范围

### 主要功能
1. 邮箱/密码登录表单
2. 基于 JWT 令牌的认证
3. 登录成功后跳转首页
4. 错误处理和用户反馈

### 文件结构
```
lib/
├── presentation/
│   └── auth/
│       ├── login_screen.dart
│       ├── login_notifier.dart
│       └── login_state.dart
├── domain/
│   ├── entities/user.dart
│   ├── usecases/login_usecase.dart
│   └── repositories/auth_repository.dart
└── data/
    ├── models/
    │   ├── user_model.dart
    │   └── login_response.dart
    ├── repositories/auth_repository_impl.dart
    └── datasources/auth_remote_datasource.dart
```

## 详细需求

### 1. 表现层 - 登录页面
- **位置**：lib/presentation/auth/login_screen.dart
- **用途**：提供用户登录 UI
- **实现内容**：
  - 使用 ConsumerStatefulWidget
  - 邮箱 TextFormField（邮箱格式验证）
  - 密码 TextFormField（8 位以上，obscureText）
  - 登录 PrimaryButton
  - 注册链接
  - 加载状态时显示遮罩
- **遵循现有模式**：使用 core/widgets/custom_text_field.dart 样式

### 2. 状态管理
- **位置**：lib/presentation/auth/login_notifier.dart
- **用途**：管理登录状态
- **实现内容**：
  - 继承 StateNotifier<LoginState>
  - login(email, password) 方法
  - 成功时保存令牌并更新状态
  - 失败时设置错误消息状态
- **遵循现有模式**：与其他 notifier 相同的模式

[...后续详细需求同上...]

## 成功标准
- [ ] 用户可输入邮箱和密码
- [ ] 点击登录按钮时调用 API
- [ ] 成功时保存令牌并跳转首页
- [ ] 失败时显示合适的错误消息（SnackBar）
- [ ] 加载中禁用按钮并显示加载指示器
- [ ] 邮箱格式和密码长度验证
- [ ] 保持现有代码风格和架构一致性
- [ ] 编写 Widget 测试（登录页面）
- [ ] 编写 Repository 测试（mock API）
- [ ] 编写 UseCase 测试

## 待确认事项
- API 端点地址是 `https://api.example.com` 吗？
- 令牌过期时需要自动刷新功能吗？
- 是否需要同时实现社交登录（Google、Apple 等）？
- 需要"忘记密码"功能吗？

---
按以上需求进行吗？有需要修改的地方请告诉我！
```

## 有效增强的技巧

### 始终请求澄清

如果项目上下文不清楚或不够：
```
如果能上传项目文件，我可以生成更准确的需求。
或者请告诉我以下信息：
- 使用的框架
- 状态管理库
- 现有项目结构
```

### 包含视觉参考

适当时提及已有页面/组件：
```
参照现有 ProfileScreen 的类似布局实现
- AppBar 样式相同
- 复用 TextFormField 设计
- 使用 PrimaryButton 组件
```

### 标注依赖关系

```
## 关联功能
- UserRepository：复用于用户信息查询
- TokenStorage：使用现有的令牌存储逻辑
- ErrorHandler：应用通用错误处理
```
